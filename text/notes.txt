The same specs (sender, nonce, payload) are reused for all three modes. build_transactions just re-signs them with different algorithms each time.
So the payloads are identical across classical, pq, and hybrid — which is the correct approach for a fair benchmark, since you're isolating the cost of the signature scheme, not the payload.




When you have raw key bytes, before you can actually use them cryptographically you have to deserialise them — parse the DER encoding, validate the elliptic curve point, and construct a Python object that the cryptography library can work with. This takes time and happens every single call in version 1.
Version 2 recognises that if you're going to be calling sign() and verify() many times, you're likely going to encounter the same keys repeatedly. So instead of throwing away the deserialised key object after each call and reconstructing it from scratch next time, it stores it in a dictionary keyed by the raw bytes.
The next time the same key bytes come in, it finds the already-constructed object in the dictionary and uses it directly, completely bypassing the deserialisation step. The actual cryptographic operation — the signing or verification itself — still happens every time, only the setup work is skipped.
In version 1, _pk_cache was declared in __init__ but never read from or written to anywhere, so it was essentially a mistake or an unfinished intention. Version 2 completes that intention for both the public and private key caches.


When you call verify() with a public key, the raw bytes have to be parsed and validated as a point on the elliptic curve before they can be used. This is the expensive part.
Without the cache, every single verification call does this parsing from scratch, even if you've seen that exact public key before.
With the cache, the first time you see a public key you parse it and store the resulting object. Every subsequent call with the same key skips the parsing entirely and goes straight to the actual signature verification using the already-built object.
Since there are only 6 authors, after the first 6 verifications the cache is fully warm and every remaining verification — across all 100 transactions — skips the parsing step completely.

Regarding commit "a710d6a7"

Both versions always store the full payload in to_dict():
python"payload": self.payload.hex()
That never changes — the raw payload is always in the transaction record.
The difference is in canonical_unsigned_bytes, which produces the bytes that get signed by the cryptographic algorithms and hashed into tx_id.
Version 1 signs:
json{"nonce": 1, "payload": "deadbeef...full hex...", "sender_id": "alice"}
The full payload is embedded inside what gets signed. If payload is 10MB, you're signing 10MB of data.
Version 2 signs:
json{"nonce": 1, "payload_hash": "a3f1c2...", "sender_id": "alice"}
Only a 64-character SHA-256 digest is embedded. The payload itself sits separately in the transaction record but is not directly in the signed bytes.
Why version 2 is still secure — if anyone tampers with the payload, the SHA-256 digest won't match anymore, so the signature verification fails. The binding is just indirect rather than direct.
Why version 2 is better practically:

Fixed-size signed data regardless of payload size
Dilithium and ECDSA both internally hash their input anyway, so version 1 is essentially double-encoding a potentially huge payload for no security gain
Cleaner separation between "what identifies the transaction" and "what the transaction carries"

The best optimisations, ranked honestly
The answer depends on your context, but if forced to rank by impact:
1. Precomputed G tables — biggest single win, essentially free
Since G never changes, you pay the precomputation once at build time and every k·G operation becomes fast table lookups forever. This touches every single signature verification. No tradeoffs, no complexity cost worth worrying about. Do this unconditionally.
2. GLV endomorphism — massive gain, secp256k1 specific
Halves the effective scalar size, which roughly halves the cost of every scalar multiplication. The endomorphism itself is nearly free to compute on secp256k1. Again, no real tradeoff. Combined with precomputed tables this is your baseline — everything else builds on top of these two.
3. Jacobian coordinates — already done for you
Any serious library handles this. Mentioning it because you'd feel it immediately if it were missing — affine arithmetic would be roughly 10× slower. Just make sure you're using a production library like libsecp256k1, not rolling your own.
4. Pippenger batching — transformative at scale
Once you're dealing with hundreds of signatures at a time (block validation, IBD, rollup sequencers), this changes the economics entirely. The O(n / log n) scaling means at n=1000 you're doing roughly 15-20% of the work of naive verification. The cost is implementation complexity and the all-or-nothing result issue.
5. Parallelism across cores — compounds everything above
Conceptually simple, practically very impactful. All the above optimisations apply per-core, so 8 cores running Pippenger batches simultaneously multiplies your throughput further. Bitcoin Core's IBD is the canonical real-world example of this working well.
6. Mempool caching — not cryptographic, but practically huge
Skipping re-verification of already-seen transactions is arguably the biggest real-world throughput gain in production blockchain nodes, but it's an engineering/caching problem rather than a cryptographic one.
